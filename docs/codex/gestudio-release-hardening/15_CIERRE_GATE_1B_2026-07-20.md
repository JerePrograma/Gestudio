## 4. Contrato financiero implementado

### 4.1 Fecha efectiva

Mensualidad:

```java
YearMonth.of(anio, mes).atDay(1)
```

Matrícula:

```java
LocalDate.of(anio, 1, 1)
```

La selección de tarifa no depende de fecha de ejecución, vencimiento, alta ni `LocalDate.now(clock)`.

### 4.2 Tarifa efectiva

Se resuelve la última tarifa con:

```text
vigenteDesde <= fechaEfectiva
```

La tarifa es obligatoria. Su ausencia aborta la operación. No existe fallback a `Disciplina.valorCuota`, `Disciplina.matricula`, cargo cero ni omisión silenciosa.

### 4.3 Condición económica

Se agregó `CondicionEconomicaServicio.vigenteOpcional(...)` sin alterar la semántica estricta de `vigente(...)`.

La condición es opcional. Si no existe, se aplica tarifa y descuento cero. Si existe:

- `costoParticular` tiene prioridad cuando no es nulo;
- se utilizan snapshots de bonificación;
- no se consulta la entidad mutable `Bonificacion` al liquidar;
- no se combina con `Inscripcion.bonificacion`.

### 4.4 Fórmula única

`LiquidacionPorVigenciaServicio` centraliza mensualidad y matrícula.

```text
porcentajeImporte = base * porcentaje / 100, escala 2 HALF_UP
descuentoImporte = porcentajeImporte + fijo, escala 2 HALF_UP
importeFinal = base - descuentoImporte, escala 2 HALF_UP
```

- importes con `BigDecimal` y escala 2;
- porcentajes con escala 4;
- `formula_version = 1`;
- importe final negativo: aborta sin persistencia parcial.

### 4.5 Matrícula multidisciplina

Por cada inscripción activa se calcula el importe final completo. Se elige:

1. mayor `importeFinal`;
2. en empate, menor ID de inscripción.

El snapshot registra tarifa y condición ganadoras y documenta el criterio. Una inscripción activa sin tarifa aborta toda la matrícula. Un alumno sin inscripciones activas no recibe matrícula cero.

### 4.6 Recargos

El `recargoId` sigue representando una regla tardía. El cargo inicial no incorpora recargo y las columnas iniciales del snapshot permanecen en cero. `RecargoServicio.aplicarRecargosAutomaticos()` continúa creando un cargo separado.

## 5. Atomicidad e idempotencia

La solución reutiliza:

- `CargoServicio`;
- `LiquidacionCargoServicio`;
- `IdempotencyLockService` indirectamente a través de los servicios existentes;
- `CargoRepositorio`;
- restricciones PostgreSQL existentes.

Garantías demostradas:

- mensualidad/matrícula, cargo y snapshot se crean dentro de una misma transacción;
- fallo inducido de snapshot revierte mensualidad y cargo;
- fallo inducido de cargo no deja mensualidad ni snapshot;
- cargo existente con snapshot se devuelve sin recalcular;
- cambio posterior de tarifa no altera el cargo ya creado;
- cargo existente sin snapshot se considera inconsistencia y no se completa con configuración actual;
- PK de `cargo_id` y bloqueos existentes impiden duplicación;
- reintentos secuenciales y concurrentes producen una sola fila por origen.

## 6. Alta de inscripción y compatibilidad API

`InscripcionServicio.crearInscripcion(...)` mantiene la transacción exterior. Si falta tarifa efectiva, se revierten:

- inscripción;
- mensualidad;
- matrícula;
- cargo;
- snapshot.

No se crea una condición económica ficticia.

Compatibilidad transicional:

- `bonificacionId` y `costoParticular` se conservan temporalmente en el DTO para deserialización;
- valores no nulos se rechazan con mensaje explícito;
- no se ignoran silenciosamente;
- frontend deja de enviarlos;
- `recargoId` se mantiene y no modifica el cargo inicial.

## 7. Frontend

Cambios operativos:

- inscripción ya no permite editar costo particular ni bonificación legacy;
